# 전기차 충전기 정기점검표 앱 개발 기준

## 프로젝트 개요

이 프로젝트는 전기차 충전기 정기점검 내역을 현장에서 바로 기록하고 관리하기 위한 Android 앱이다. 집중관리 대상 충전기의 이상 유무를 매월 확인하고, 점검 결과를 누락 없이 기록하여 운영 관리 효율을 높이는 것을 목표로 한다.

## 개발 배경

- 채비관제 운영 충전기의 운영 수익 극대화를 위해 집중관리 대상 충전기를 체계적으로 관리한다.
- 상위 매출 약 45%에 해당하는 충전기를 우선 관리 대상으로 삼는다.
- 집중관리 대상 충전기는 매월 1회 정기점검을 실시한다.
- 정기점검을 통해 충전기의 이상 유무를 확인하고 점검 결과를 기록한다.
- 별도 종이 점검표나 사후 입력 업무를 줄이고, 앱에서 현장 즉시 기록이 가능하게 한다.

## 핵심 사용자 흐름

- 사용자는 집중관리 대상 충전기 목록을 확인한다.
- 사용자는 충전기를 선택해 해당 월의 정기점검표를 작성한다.
- 사용자는 점검 항목별 정상, 이상, 해당 없음 등의 상태를 입력한다.
- 사용자는 이상 사항이 있을 경우 메모나 필요한 증빙 정보를 남긴다.
- 사용자는 점검 결과를 저장하고, 이후 충전기와 점검월 기준으로 기록을 조회한다.

## 기술 방향

- 아키텍처는 클린 아키텍처를 기본으로 한다.
- View는 Jetpack Compose를 사용한다.
- 화면 이동은 Navigation 3를 사용한다.
- 네트워크는 Retrofit, OkHttp, Kotlinx Serialization 조합을 기본으로 한다.
- 의존성 주입은 Dagger Hilt를 기본으로 사용한다.
- 개발은 테스트 코드 중심으로 진행한다.
- Android 프레임워크가 필요 없는 업무 규칙은 JVM 단위 테스트로 검증한다.
- 앱은 Play Store에 배포하지 않고 내부 APK 형태로 배포한다.

## 배포 원칙

- 배포 대상은 내부 사용자로 제한한다.
- Play Store 배포를 전제로 한 기능이나 정책 대응은 우선순위에서 제외한다.
- 배포 산출물은 APK를 기본으로 한다.
- 내부 배포용 release 빌드는 별도 서명 키로 관리한다.
- 버전 코드는 내부 배포 이력을 추적할 수 있도록 배포마다 증가시킨다.
- APK 파일명에는 앱 이름, 버전명, 빌드 타입, 배포일을 포함한다.
- 내부 사용자가 설치 전 신뢰할 수 있도록 배포 경로와 설치 안내를 명확히 한다.
- 업데이트 방식은 기존 APK를 새 APK로 교체 설치하는 흐름을 기본으로 한다.
- 운영 데이터 손실을 막기 위해 앱 업데이트 시 로컬 데이터 유지 여부를 확인한다.

## 아키텍처 원칙

- 의존성 방향은 `Presentation -> Domain <- Data`를 따른다.
- `Domain` 계층은 앱의 핵심 업무 규칙을 담고 Android 프레임워크에 의존하지 않는다.
- `Presentation` 계층은 Compose 화면, Navigation 3 화면 이동, ViewModel, UI 상태, 사용자 이벤트 처리를 담당한다.
- `Data` 계층은 로컬 DB, 원격 API, 파일 저장소 등 실제 데이터 입출력을 담당한다.
- Repository 인터페이스는 `Domain` 계층에 두고 구현체는 `Data` 계층에 둔다.
- UseCase는 하나의 명확한 업무 동작만 담당한다.
- Entity, UseCase, Repository 인터페이스에는 UI나 저장 방식의 세부 구현을 넣지 않는다.
- Mapper를 사용해 `Data` 모델과 `Domain` 모델을 분리한다.
- 화면에서 DB, API, DataSource를 직접 호출하지 않는다.

## 의존성 주입 원칙

- 의존성 주입은 Dagger Hilt를 사용한다.
- 앱 진입점은 `@HiltAndroidApp`이 적용된 Application 클래스를 사용한다.
- Activity는 Hilt 주입이 가능하도록 `@AndroidEntryPoint`를 적용한다.
- ViewModel은 `@HiltViewModel`과 `@Inject constructor`를 기본으로 작성한다.
- Repository 구현체, DataSource, Retrofit, OkHttp 등 외부 의존성은 Hilt Module에서 제공한다.
- Domain 계층은 Hilt에 직접 의존하지 않는다.
- 테스트에서는 필요한 경우 Fake 또는 Test Module로 의존성을 교체한다.

## 화면 구성과 이동

- 신규 화면은 Jetpack Compose로 작성한다.
- XML layout은 신규 화면에 사용하지 않는다.
- 화면 이동은 Navigation 3를 기본으로 사용한다.
- Navigation route와 화면 인자는 타입 안정성을 우선한다.
- 각 View는 `Route`와 `Screen`을 구분해서 작성한다.
- `Route`는 화면 진입점으로 Navigation 인자, ViewModel 연결, 상태 수집, 화면 이동 이벤트 연결을 담당한다.
- `Screen`은 순수 UI로 작성하고 상태와 이벤트 콜백을 파라미터로 전달받는다.
- 각 View의 UI 업데이트는 ViewModel이 제공하는 `StateFlow` 기반 `UiState`를 통해 처리한다.
- `Route`는 `StateFlow`를 lifecycle-aware 방식으로 수집해서 `Screen`에 전달한다.
- `Screen` 내부에서 장기 유지되어야 하는 화면 상태를 직접 `remember`로 관리하지 않는다.
- 화면 간 데이터 전달은 최소화하고 필요한 식별자만 전달한다.
- 상세 데이터는 전달받은 식별자를 기준으로 ViewModel에서 다시 조회한다.
- Composable은 화면 표시와 사용자 이벤트 전달에 집중한다.
- Composable에서 UseCase, Repository, DataSource를 직접 호출하지 않는다.
- 화면 상태는 ViewModel이 관리하고 Composable은 상태를 구독해서 그린다.
- 일회성 이벤트는 일반 UI 상태와 분리해서 처리한다.

## 네트워크 원칙

- API 정의는 Retrofit interface로 작성한다.
- 공통 헤더, 인증, 로깅, 타임아웃, 재시도 정책은 OkHttp interceptor에서 처리한다.
- JSON 직렬화와 역직렬화는 Kotlinx Serialization을 사용한다.
- 점검 저장 API에 전달할 데이터는 충전기 식별자, 점검월, 점검자, 생성 시각, 이상 사항 메모, 전체 점검 그룹과 항목별 선택 상태를 하나의 payload로 통합한다.
- 아직 선택되지 않은 항목은 서버 전송 형식에서 누락하지 않고 `NOT_SELECTED` 상태로 포함한다.
- Retrofit DTO는 `Data` 계층에만 둔다.
- Repository 구현체는 RemoteDataSource 호출 결과를 Domain 모델로 변환해서 반환한다.
- 네트워크 오류, 서버 오류, 파싱 오류는 Domain 또는 Presentation에서 처리 가능한 형태로 변환한다.
- ViewModel이나 Composable에서 Retrofit API를 직접 호출하지 않는다.

## 엑셀 점검표 데이터 원칙

- 점검항목 데이터는 앱 실행 중 설정 화면에서 엑셀 파일을 다운로드해 갱신한다.
- 다운로드한 엑셀 파일은 앱 내부 저장소에 보관한다.
- 엑셀 파일 형식은 고정된 정기점검표 양식을 기준으로 한다.
- 앱은 저장된 엑셀 파일을 읽어 점검항목 리스트, 청소 기준, 청소방법 사례 데이터를 구성한다.
- 엑셀 파싱 로직은 `Data` 계층의 DataSource에 둔다.
- 화면과 ViewModel은 엑셀 파일을 직접 읽지 않는다.
- Repository는 엑셀 파싱 결과를 Domain 모델로 변환해 반환한다.
- 고정 양식의 셀 위치나 행 범위가 바뀌면 파서 테스트를 먼저 수정한다.
- 엑셀 다운로드 실패 시에는 기존에 저장된 파일을 우선 사용한다.
- 기존 파일도 없을 경우 화면에서 데이터 없음 상태를 명확히 보여준다.

## 테스트 코드 중심 개발

- 기본 개발 방식은 테스트 코드를 먼저 작성하거나 기능 구현과 함께 작성하는 방식으로 진행한다.
- 업무 규칙은 `Domain` 계층 테스트로 먼저 검증한다.
- UseCase는 입력, 정상 결과, 예외 상황을 단위 테스트로 확인한다.
- ViewModel은 UI 상태 변경과 사용자 이벤트 처리 결과를 테스트한다.
- Repository 구현체는 로컬 DB, 원격 API, 캐시 동작을 분리해서 테스트한다.
- Android 프레임워크가 필요 없는 로직은 JVM 단위 테스트로 작성한다.
- 화면 동작이나 실제 Android 컴포넌트가 필요한 경우에만 Instrumented Test를 사용한다.
- 버그를 수정할 때는 재현 테스트를 먼저 추가한 뒤 수정한다.
- 테스트하기 어려운 코드는 구조를 다시 검토한다.

## 코드 작성 원칙

- Kotlin과 Android 공식 스타일을 따른다.
- 변경은 작게 만들고 한 번에 하나의 목적만 다룬다.
- 기존 구조와 네이밍을 우선 따르고 새 패턴은 꼭 필요할 때만 추가한다.
- 화면 로직, 업무 규칙, 데이터 입출력 구현을 섞지 않는다.
- 사용자에게 보이는 문구는 짧고 명확하게 작성한다.
- 현장 사용자가 빠르게 점검하고 저장할 수 있는 흐름을 우선한다.
- 중복 로직은 두 번 이상 반복될 때 공통 함수나 클래스로 분리한다.
- 널 처리와 예외 처리는 호출부에서 동작을 예측할 수 있게 명시한다.
- 임시 코드, 사용하지 않는 파일, 디버그 로그는 남기지 않는다.

## UI 작성 원칙

- 체크시트 사용 흐름을 방해하지 않도록 화면은 단순하게 유지한다.
- 월 1회 정기점검 업무가 빠르게 끝나도록 입력 단계와 화면 이동을 최소화한다.
- 버튼, 입력 필드, 상태 표시는 역할이 바로 드러나는 이름과 배치를 사용한다.
- 점검 완료, 이상 있음, 미점검 상태는 명확히 구분해서 보여준다.
- API 통신, 초기 데이터 로딩, 저장 처리 중에는 공통 로딩 View를 사용한다.
- 공통 로딩 View는 사용자가 임의로 닫을 수 없는 Dialog로 표시한다.
- 작은 화면에서도 텍스트가 잘리지 않도록 확인한다.
- 접근성을 위해 주요 UI 요소에는 적절한 라벨을 제공한다.

## 검증 원칙

- 가능한 경우 테스트를 먼저 실행하고 실패를 확인한 뒤 구현한다.
- 구현 후 관련 단위 테스트와 빌드 검증을 실행한다.
- 빌드와 테스트 실행은 개발 작업의 기본 검증 단계로 보고, 별도 확인 없이 필요한 시점에 실행한다.
- 빌드나 테스트가 실패하면 원인을 확인하고 수정한 뒤 다시 검증한다.
- Hilt/KAPT 생성 코드 충돌을 피하기 위해 Gradle 빌드와 테스트 검증은 순차 실행한다.
- 변경한 기능은 정상 흐름과 예외 흐름을 함께 확인한다.
- 사용자 데이터가 추가, 수정, 삭제되는 기능은 저장 결과까지 확인한다.
- 점검 기록이 충전기, 점검월, 점검자 기준으로 정확히 남는지 확인한다.
- 버그 수정 시에는 같은 문제가 다시 생기지 않도록 재현 조건을 기록한다.
